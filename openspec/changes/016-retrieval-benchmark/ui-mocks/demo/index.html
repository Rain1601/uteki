Retrieval Benchmark

管道保真度回归测试。 用法、不用法、CI 门禁、报告页 — 全部走一遍。

✓ in scope · measured here retrieval 准不准 · citation 数字对不对 · future leakage 有没有 · abstention 答不出时承认了吗 · contradiction 浮现了对方观点吗
✗ out of scope · do NOT measure here PnL · hit rate · agent 的 BUY 跑赢 SPY 了吗 · conviction calibration · 文风好不好。 这些一律不进 CI 门禁。

最终目标 · 一个开发者的视角

想象 016 完工以后,你刚改了 services/api/src/uteki_api/tools/web_search.py(加了某条新 fallback 路径), 正准备 push。 接下来你会做的事 — 每一步对应一个 demo 页:

1
本地跑 benchmark
$ ./scripts/benchmark-run.sh --case-set approved --agent company_research_pipeline · 5-15 分钟 · 看终端输出
CLI 2
报告页 — 看到红
5 cases × 5 scorers 矩阵 · CI 门禁: FAIL · lookahead_leakage = 1 (一条 future_trap 被引用了)
FAIL 3
钻进失败的那条 case
看 case YAML · 对照 corpus 文档 · 看 agent 实际答案 · 看每个 scorer 怎么打的分
drill-down 4
对照 main · 看是什么变了
main 的 baseline report vs 你这条 branch 的 report · 5 个 scorer 哪个动了 · 哪条 case 动了
diff
没有第 5 步。 你不需要打开浏览器看 dashboard,也没有"15 分钟内 agent 又跑了一遍"的等待。 整个 loop 在你 push 之前完成。 这是跟 015 eval-workbench 最大的不同 — 015 是主观打分(更慢、更贵、不进 CI),016 是客观检查(快、零成本、进 CI)。
